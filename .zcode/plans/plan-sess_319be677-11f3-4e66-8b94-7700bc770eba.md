# Полная многоязычность + фикс ошибки сборки `/certificates`

## 0. Корневая причина ошибки сборки

В проекте **дублирующаяся структура маршрутов**: страницы существуют и ВНЕ `[locale]` (`src/app/page.tsx`, `src/app/certificates/page.tsx`, `src/app/blog/...`), и ВНУТРИ (`src/app/[locale]/page.tsx`, `src/app/[locale]/certificates/page.tsx`). Только `[locale]/layout.tsx` оборачивает детей в `NextIntlClientProvider`. Страницы вне `[locale]` остаются **без провайдера** → `useTranslations()` бросает исключение при prerender → сборка падает именно на `/certificates` (т.к. `CertificateCard` в detailed-режиме вызывает `useTranslations('Common')`, а `/blog` не падает только потому, что его компоненты не вызывают `useTranslations`).

**Решение:** консолидировать **всё** в `[locale]`. Удалить дубликаты вне `[locale]`. Тогда провайдер один на всё приложение.

## 2. Способ перевода данных API (выбранный пользователем подход)

Переводимые поля в `data.ts` становятся объектами **`{uz, ru, en}`**. Типы в `src/types/index.ts` обновляются: вводятся generic-типы `Localized = {uz:string; ru:string; en:string}` и `LocalizedData<T>` (где переводимые строковые поля → `Localized`, а структурные поля: id, hue, github, image, icon, isPublic, featured, value, suffix — без изменений).

**API-роуты** (`/api/about|projects|services|certificates|skills|contact`) читают локаль из заголовка `x-locale` (его выставляет middleware для всех `/api/*` запросов — добавлю это в `proxy.ts`) или query `?lang=`, и возвращают «плоские» данные уже на нужном языке (преобразование `{uz,ru,en}` → строку).

**`useData` хук** (`src/hooks/useData.ts`): добавляется чтение текущей локали через `useLocale()` из next-intl; в `fetch` добавляется заголовок `x-locale: <locale>`; ключ `useEffect` включает `locale` → **при смене языка данные автоматически перезагружаются**.

**Почему этот способ (как просил пользователь — «самый оптимальный и без ошибок в будущем»):**
- один источник истины (`data.ts`), нет дублирования между `data.ts` и `messages/*.json`
- TypeScript **гарантирует** наличие всех 3 языков для каждого поля → невозможно оставить частичный перевод
- API возвращает уже готовые данные — компоненты не меняют логику, только отображают `data.title` как раньше
- смена языка = автоматическая перезагрузка данных через эффект в `useData`

**Имена собственные (как ответил пользователь — «только описания»):** названия проектов, названия категорий навыков, issuer сертификатов, имена технологий — **остаются как есть**. Переводятся только: описания, подзаголовки, body, lead, role/year/description таймлайна, label статистик, label funFacts, contact.description.

## 5. Обновление middleware (`src/proxy.ts`)

Сейчас `matcher` исключает `/api` из обработки next-intl → API не знает локаль. Добавить в middleware проброс локали в заголовок `x-locale` для **api-запросов** (через `request.headers` → `NextResponse.next()` с `request.headers.set`), чтобы `useData` автоматически слал нужный язык, а API-роуты читали `headers.get('x-locale')`.
