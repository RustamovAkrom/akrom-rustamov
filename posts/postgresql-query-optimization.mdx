Database performance is often the bottleneck in backend applications. Here is how I approach PostgreSQL optimization.

## Analyzing Slow Queries

Start with `EXPLAIN ANALYZE`. It shows the actual execution plan and timing. Look for:

- **Sequential scans** on large tables — add indexes
- **Nested loops** with high row counts — consider hash joins
- **High buffer reads** — check cache hit ratio

## Indexing Strategies

- **B-tree indexes** — default, great for equality and range queries
- **Partial indexes** — index only a subset of rows, smaller and faster
- **Composite indexes** — cover multi-column queries, order matters
- **GIN indexes** — for JSONB, arrays, and full-text search

## Query Patterns to Avoid

- `SELECT *` — fetch only needed columns
- N+1 queries — use JOINs or batch fetching
- Functions on indexed columns — prevents index usage
- Unbounded pagination — use keyset pagination with `cursor`

## Connection Pooling

Use PgBouncer in transaction mode. It reduces connection overhead and allows thousands of concurrent clients with a small pool of actual database connections.
